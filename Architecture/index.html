1. Introduction and Goals

1.1 Requirements Overview

The Game Logic System is the core engine for a medieval-themed game emphasizing systemic interactions and emergent gameplay. It manages:

  • Game state and progression

  • Location-based actions and interactions

  • Character relationships and knowledge

  • Resource and energy management

  • Time and schedule systems

2. Architecture Constraints

2.1 Technical Constraints

  • Strongly-typed C# implementation

  • No direct UI dependencies

  • Pure game logic separated from content generation

  • Deterministic outcomes for all actions

  • Use of enums and immutable data structures where appropriate

  • Adherence to Domain-Driven Design principles (Entities and Value Objects)

2.2 Organizational Constraints

  • Must support external content creation pipeline

  • Must facilitate automated testing

  • Must enable clear separation between content and mechanics

4. Solution Strategy

The system employs several key strategic decisions:

  1. State Isolation: GameState class contains all mutable game state with no external dependencies.

  2. Clean Actions: GameManager handles all state modifications through well-defined actions.

  3. Content Independence: Location and character systems are populated at startup but operate independently.

  4. Deterministic Design: All game mechanics produce predictable outcomes based on the current state.

  5. DDD Principles: Locations and Action Templates are treated as Entities, while Location Spots and Action Implementations are treated as Value Objects.

6.3 State Change Processing

The system uses a two-phase state change approach:

  1. Pending Changes Phase:

    • Changes are collected in ActionResultMessages.

    • Multiple outcomes can be queued.

    • No state is modified.

  2. Application Phase:

    • Changes are applied atomically.

    • Results are validated.

    • Changes generate new messages.

    • Process repeats until no new changes.

This ensures consistent state transitions and proper change propagation.

7. Cross-cutting Concepts

7.1 Domain Models

Core domain models include:

  • Actions (INVESTIGATE, LABOR, DISCUSS, etc.)

  • Knowledge Flags (binary states that unlock capabilities)

  • Energy Types (Physical, Social, Focus)

  • Time Windows (Morning, Afternoon, Evening, Night)

  • Location Archetypes (TAVERN, MARKET, FOREST, etc.)

  • Location Spots (Bar, Tables, Fireplace, etc.)

  • Action Templates (with Basic Action Types)

  • Action Implementations (Value Objects created on the fly)

7.2 State Management

  • GameState is the single source of truth.

  • All state modifications go through GameManager.

  • State changes are atomic and deterministic.

  • External systems can only read state.

  • State is structured to support efficient queries and updates.

7.3 Exception Handling

  • Validation errors returned as Results (e.g., using a Result<T> type).

  • No exceptions for expected failure cases (e.g., insufficient resources).

  • System exceptions only for true errors (e.g., null references, unexpected data).

7.4 Content Creation Pattern

The system employs a comprehensive builder pattern for game content:

  • LocationBuilder: Constructs location definitions (Entities).

  • CharacterBuilder: Creates character definitions (Entities).

  • ActionTemplateBuilder: Defines available action templates (Entities) with Basic Action Types.

  • NarrativeBuilder: Constructs narrative sequences.

  • LocationContextBuilder: Creates Location Contexts (Value Objects)

This provides:

  • Type-safe content creation.

  • Consistent validation.

  • Flexible composition.

7.5 Action Implementation Generation

Action implementations (Value Objects) are generated on demand by the Action Implementation Generator when the player interacts with a location. The generator uses:

  • The Location (Entity) and its current Context (Value Object).

  • The Action Template (Entity) selected by the player.

  • The Location Spot (Value Object) the player is interacting with.

  • Cost and Reward rules based on the location’s Type and the Action Template’s Basic Action Type.

This ensures that:

  • Only valid actions are available in a given location and context.

  • Costs and rewards are contextually appropriate.

  • Action implementations are transient and do not need to be persisted.

8. Architectural Decisions

8.1 Strict State Isolation

GameState has no dependencies on other systems to ensure clean separation and testability.

8.2 Content Independence

Location and character systems are populated at startup but operate independently of content generation.

8.3 Deterministic Design

All game mechanics produce predictable outcomes based on the current state for testing and debugging.

8.4 Energy Management

The game features distinct energy types:

  • Physical Energy: Manual labor and physical tasks.

  • Focus Energy: Investigation and observation.

  • Social Energy: Character interactions.

Each energy type:

  • Has a maximum capacity.

  • Regenerates through rest.

  • Is consumed by specific actions.

  • Can receive time window bonuses/penalties.

8.5 Game Initialization

The system uses a GameSetup class to create consistent initial states:

  • Starting resources (health, energy, coins).

  • Initial location setup.

  • Basic action availability.

  • Starting inventory capacity.

  • Default skill levels.

8.6 Resource System

Resources are managed through:

  • Fixed or dynamic inventory slots.

  • Resource types enumeration.

  • Add/remove validation.

  • Capacity tracking.

  • Resource transformation rules (e.g., combining herbs to create potions).

11. Glossary

Term Definition

GameState

Central class holding all mutable game state.

GameManager

Coordinates all state modifications through defined actions.

LocationSystem

Manages definitions and rules for game locations (Entities).

CharacterSystem

Manages definitions and rules for game characters (Entities).

Knowledge Flag

Binary state indicating specific player knowledge.

Energy Type

Resource used for performing actions (Physical, Social, Focus).

Action Template

Defines the basic structure and logic of an action (Entity).

Action Implementation

A concrete instance of an action, generated on the fly (Value Object).

Location Spot

A specific area within a location where actions can be performed (Value Object).

Location Archetype

A template for a type of location, defining typical activities and resources (Entity).
